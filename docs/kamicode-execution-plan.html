KamiCode Incremental Execution Plan

A concrete, testable rollout plan for turning the current T3 Code fork into KamiCode: desktop clients run local agents, web/mobile runs VPS agents, GitHub remains the source of truth, shared threads can be handed off to team/VPS environments, and workspaces can contain one or many repositories.

Rules For Every Step

  • Each step must be independently testable.
  • No step depends on a later step to prove it works.
  • After each step: run typecheck, relevant tests, and one manual acceptance check.
  • Desktop mode always means local agent execution.
  • Web/mobile mode always means VPS agent execution.
  • Shared threads are portable handoff packages, not live migration of a running Codex or Claude Code process.
  • Workspaces may contain one or many GitHub repositories.

Core Defaults

Code Sync

GitHub is canonical. VPS and local machines sync through Git.

Project Shape

A KamiCode workspace can contain one repo or many repos.

Desktop

Always runs Codex, Claude Code, or other agents locally on the employee PC.

Web/Mobile

Always runs agents on the VPS against VPS-managed clones/worktrees.

Shared Threads

A shared thread syncs conversation state, repo delta, untracked files, and project memory so a teammate or web/mobile VPS agent can continue it.

Access

GitHub OAuth and GitHub App permissions decide user and repo access.

Credentials

Shared provider credentials never go to browser-only clients.

Step 1

Rebrand Visible App Text

Change: Rename visible product text from T3 Code to KamiCode.

Do not change: package names, internal module names, database schemas, or behavior.

Test

  • Run bun run typecheck.
  • Start the app and confirm visible UI says KamiCode.
  • Confirm existing startup still works for server, web, and desktop.
Step 2

Add Workspace Contract Types

Change: Add shared contract schemas for multi-repo workspaces.

Types

  • KamiWorkspace
  • KamiWorkspaceRepository
  • KamiWorkspaceId
  • GitHubRepositoryRef

Test

  • Add schema encode/decode tests.
  • Verify one workspace can contain one repo.
  • Verify one workspace can contain many repos.
  • Run bun run typecheck.
Step 3

Add Workspace Database Tables

Change: Add migrations for workspaces and workspace_repositories.

Test

  • Migration creates tables on a fresh database.
  • Migration is safe on an existing database.
  • Insert and read a workspace with multiple repos in a persistence test.
Step 4

Create Workspace Repository Service

Change: Add server-side service for creating, listing, reading, updating, and archiving workspaces.

Test

  • Create workspace with one repo.
  • Create workspace with multiple repos.
  • Reject duplicate repo inside the same workspace.
  • Archive workspace and confirm it no longer appears in the active list.
Step 5

Expose Workspace RPC Methods

Change: Add WebSocket RPC methods for workspace list, detail, create, update, and archive.

Test

  • RPC list returns an empty array on fresh install.
  • RPC create returns the created workspace.
  • RPC detail returns all repos in the workspace.
  • RPC archive hides the workspace from the active list.
Step 6

Add Basic Workspace UI

Change: Add UI screen for workspace list and workspace detail.

No GitHub auth yet.

Test

  • Create a Hermes workspace manually through UI/dev path.
  • Show multiple repos under Hermes.
  • Open workspace detail page directly by URL.
  • Mobile viewport shows readable workspace list and repo list.
Step 7

GitHub OAuth Login

Change: Add GitHub OAuth login and map GitHub users to KamiCode users.

Test

  • Unauthenticated user sees login gate.
  • GitHub OAuth callback creates or updates a user record.
  • Session state includes GitHub login and user id.
  • Logout clears session.
Step 8

GitHub App Installation

Change: Add GitHub App installation support for the company org.

Test

  • Admin can see GitHub App installation status.
  • Server can list repositories from the installation.
  • Webhook signature verification rejects invalid signatures.
  • Repository metadata sync updates local cache.
Step 9

GitHub-Based Workspace Access

Change: Compute workspace access from GitHub repo permissions.

Rule: A user can see a workspace if they can access at least one repo in it.

Test

  • User with access to all Hermes repos sees all repos.
  • User with access to only one Hermes repo sees Hermes but only that repo as usable.
  • User with access to no Hermes repos does not see Hermes.
  • Server rejects direct RPC access to unauthorized repos.
Step 10

Desktop Local Workspace Clone

Change: Add “Clone Workspace” flow to desktop.

Behavior: Desktop clones authorized repos into a local workspace folder.

Test

  • User selects local base directory.
  • Hermes workspace clones multiple authorized repos.
  • Unauthorized repos are skipped and clearly marked.
  • Generated local manifest maps each repo to its local path.
Step 11

Enforce Desktop Local Agent Execution

Change: Ensure desktop workspace threads use local Codex/Claude Code only.

Test

  • Start a desktop thread and confirm provider process starts on the employee PC.
  • Confirm VPS is not executing shell commands for desktop-local threads.
  • Agent can edit files in the local cloned repo.
  • Agent can run local tests/commands from the local repo path.
Step 12

VPS Managed Workspace Clones

Change: VPS stores managed clones for web/mobile execution.

Behavior: Web/mobile opens the VPS copy, not employee local files.

Test

  • VPS clones Hermes repos into managed storage.
  • Web thread runs commands on VPS paths.
  • Desktop thread runs commands on local paths.
  • Same workspace can be opened from desktop and web with different execution locations.
Step 13

Shared Thread Handoff

Change: Add a thread-level Share thread action that creates a portable handoff package for VPS/team continuation.

Behavior: Local provider processes are not migrated. KamiCode persists enough canonical state for a new provider runtime on the VPS to continue the same thread.

Captured State

  • Thread metadata, model/provider settings, transcript, and turn history.
  • Workspace/repo identity, branch, base commit, staged state, and tracked changes.
  • Untracked file bundle with checksums and explicit ignored-file exclusions.
  • CAMI project memory snapshot with source thread, turn id, author, and timestamp.
  • Source environment metadata so the receiver knows the handoff origin.

Test

  • Right-clicking a thread exposes Share thread.
  • Sharing uploads transcript, repo patch, untracked files, and project memory.
  • VPS applies the package to a managed clone/worktree and verifies checksums.
  • A different user can continue the shared thread from web/mobile on the VPS.
  • If the source PC goes offline after sync, the shared VPS thread remains usable.
  • Ignored files and likely secrets are excluded unless explicitly approved.
Step 14

Mobile-First Web Thread UI

Change: Make web/mobile usable for remote VPS work.

Scope: threads, approvals, diffs, status, quick replies.

Test

  • Phone viewport can open workspace and thread.
  • User can send a prompt from phone.
  • User can approve or reject tool/command request from phone.
  • Diff view is readable on phone.
Step 15

Shared Provider Credential Bundle Storage

Change: Add admin-managed shared provider credential bundles.

Rule: Browser clients never receive raw provider credentials.

Test

  • Admin can create credential bundle metadata.
  • Credential payload is encrypted at rest.
  • Browser credential request is rejected.
  • Audit log records bundle creation without secret values.
Step 16

Trusted Desktop Device Registration

Change: Desktop registers as a trusted device with a device key.

Test

  • Fresh desktop creates device identity.
  • Server stores device public key and user mapping.
  • Revoked device cannot request credential payload.
  • Device list appears in admin UI.
Step 17

Desktop Credential Sync To OS Vault

Change: Trusted desktop can receive encrypted shared provider credentials and store them in the OS vault.

Test

  • Trusted desktop receives encrypted payload.
  • Payload decrypts only on that device.
  • Credential is stored in OS vault, not plain config.
  • Local Codex/Claude Code can start using synced credential material.
Step 18

Offboarding Flow

Change: Admin can remove an employee.

Behavior: Revoke KamiCode sessions, revoke devices, block credential refresh, and refresh GitHub-derived access.

Test

  • Removed employee cannot log into web.
  • Removed employee desktop cannot sync workspaces after reconnect.
  • Removed employee desktop cannot refresh shared credentials.
  • Admin sees warning that offline synced credentials require provider credential rotation.
Step 19

Action Audit Log

Change: Add audit log for important activity.

Events: login, device registration, workspace open, clone, agent turn, command approval, git action, credential grant/revoke.

Test

  • Each event stores user, device, workspace, repo, and timestamp when available.
  • Secrets and bearer tokens are redacted.
  • Admin can filter audit log by user.
  • Admin can filter audit log by workspace.

First Concrete Task

Start with Step 1: Rebrand Visible App Text. It is small, reversible, and proves the fork can build and run before touching auth, GitHub, workspaces, or credentials.
