KamiCode product and implementation plan

Persistent Remote Runtimes for VPS-Backed Agent Work

This plan defines the target experience where a user can connect a VPS from the local KamiCode GUI, let KamiCode provision itself remotely, select that runtime from the composer, and keep agents running after the local PC is closed.

Primary user: desktop KamiCode user Runtime target: Linux VPS over SSH Core promise: survives local PC shutdown Project memory path: .camie/project-memory.md
1. UX Plan The user-facing flow, product behavior, states, and guardrails. 2. Build Plan Exact implementation slices, data model, APIs, provisioning, and tests.

Section 1

UX Plan

The remote runtime should feel like a normal place to run work, not like a network debugging project. The user gives KamiCode VPS access once. KamiCode handles install, remote service management, connection setup, provider/profile migration where safe, and project porting when the runtime is first used.

Product promise

Connect a VPS, choose it from the composer, and let agents keep working there even if this computer closes.

This promise only holds when the VPS owns the running KamiCode service and reachable endpoint. A local desktop SSH tunnel can be used for setup, but it cannot be the final access path.

Primary User Journey

Open Connections

User opens Settings -> Connections and selects Add remote runtime.

Choose VPS over SSH

User selects a runtime type named VPS over SSH. The screen explains that SSH is used to provision the host, but the resulting runtime runs independently on the VPS.

Enter SSH credentials

User enters host, port, username, and either password or private key. Passwords are used for this setup attempt unless the user explicitly saves a credential.

Preflight and install

KamiCode checks OS, CPU architecture, disk space, shell, git, curl, Node availability, service support, and network requirements. It then installs or updates the remote runtime.

Configure persistent access

KamiCode creates a VPS-owned reachable endpoint using a managed relay or tunnel. The connection must keep working after the desktop shuts down.

Sync provider profiles

KamiCode offers provider profile sync. Safe settings are synced by default. Secrets and auth folders are opt-in and validated after transfer. Providers that cannot be moved cleanly show Needs login on VPS.

Use runtime from composer

The runtime appears in the composer checkout/runtime selector next to Local checkout. User selects it and sends a message.

Automatic project porting

If the selected project has never run on that VPS, KamiCode prepares a remote checkout first, including source code, current branch state, dirty changes, AGENTS/instructions, and .camie/project-memory.md.

Agent continues remotely

The agent session runs on the VPS. Closing the local PC disconnects the client, not the runtime.

Connection Wizard Screens

1. Runtime Type

Options: Local backend, Pair existing backend, VPS over SSH. This plan implements the third as a persistent runtime, not a temporary SSH tunnel.

2. SSH Access

Fields: host, username, port, password, private key, passphrase, label. Include discovered SSH config hosts, but always merge typed fields into selected hosts.

3. Preflight

Show checks as rows: reachable, Linux, arch, disk, writable home, shell, git, curl, Node, service manager, network endpoint, provider profile compatibility.

4. Installation

Show a log-style progress surface with bounded, named stages. Keep raw logs behind View details.

5. Provider Profiles

Let the user choose which profiles to transfer. Default to settings and model preferences. Require explicit opt-in for auth folders and tokens.

6. Finish

Show runtime label, endpoint, health, installed version, provider status, and a primary action: Use in composer.

Runtime Card States

State Meaning Primary action
Provisioning SSH connected; installation or endpoint setup is running. View progress, cancel.
Ready Runtime is paired, reachable, healthy, and usable from the composer. Use in composer.
Needs Attention Runtime works, but one or more providers need login or a project needs sync. Review provider status.

Remote Update UX

The VPS runtime needs to keep up with KamiCode releases without requiring the user to SSH in manually. Updates should be visible, reversible, and tied to the runtime's selected channel.

Channel choice

Each runtime has an update channel: latest, nightly, or pin current version. Default to the same channel as the desktop unless the user overrides it.

Update prompts

Runtime cards show Update available with current version, target version, release channel, and whether the update is required for compatibility.

Automatic policy

Let users choose Manual, Auto-update when idle, or Match desktop channel. Never interrupt an active agent turn unless the update is explicitly forced.

Rollback

Keep the previous remote runner package/version and service config. If health checks fail after update, rollback automatically and mark the runtime as needing attention.

Update states users should see
  • Up to date: remote version matches selected channel target.
  • Update available: update is optional and can be scheduled.
  • Update required: local client and remote server protocol versions are incompatible.
  • Updating: service is installing the new runner and will restart when idle.
  • Rollback complete: update failed, previous version is running again.
  • Update failed: service could not install or restart; show logs and repair.

Composer Runtime Selector

The composer should make execution location visible every time a user starts or continues work. If the current UI already has a local checkout dropdown, the persistent VPS runtime should appear in that same control.

Selection Meaning Before send After send
Local checkout Run agent on this machine against the local project. No remote setup. Existing local behavior.
VPS: Production Box Run agent on the remote VPS runtime. Check runtime health and project-port mapping. Remote thread starts on the VPS and streams back to the local UI.
VPS: Production Box - first use Runtime exists, but project has not been initialized there. Run project porting flow before dispatching the first turn. Save project mapping and start the remote turn.

Project Porting UX

Trigger

The first time a user sends a message with a remote runtime selected for a local project, KamiCode runs Prepare remote checkout before starting the agent.

What the user should see during project porting
  1. Detect project state: git repo, branch, HEAD SHA, dirty files, untracked files.
  2. Show a compact progress row above the composer: Preparing remote checkout...
  3. Create or update remote project directory under ~/kamicode-projects/.
  4. Clone or sync source files.
  5. Copy required context: AGENTS.md, selected instructions, and .camie/project-memory.md.
  6. Apply local dirty changes as a patch or file overlay.
  7. Register the remote project with the remote KamiCode backend.
  8. Start the user message after porting completes.
Conflict and safety UX
  • If remote checkout has uncommitted changes, pause and show Remote checkout has local changes.
  • If local project has secret files that match denylisted patterns, exclude them and show Secrets were not copied.
  • If the repo has no remote, use archive sync and label the mapping as File sync.
  • If the project is very large, show an estimate and allow cancel before upload.
  • If dependencies are missing remotely, let the agent or project setup script handle them after checkout exists.

Provider Profile UX

The desired outcome is "do not make me reauthenticate if avoidable." The safe product design is "try profile migration, verify it, and clearly fall back to reauth when a provider refuses moved credentials."

Item Default Reason User copy
KamiCode provider settings Copy Needed for model/profile parity. Copied from local settings.
Provider auth folders such as .codex Ask May contain sensitive tokens and machine-bound auth. Copy encrypted over SSH and verify. You may still need to sign in again.
GitHub CLI auth Ask Useful for PRs, but highly sensitive. Copy GitHub CLI auth to let remote agents push branches and open PRs.
.env files and private keys Do not copy Project secrets should not move silently. Secrets are excluded unless you explicitly add them.

Important guardrail

Copying the entire .codex or other provider home can work in some cases, but it must be opt-in and verified. Some providers bind credentials to the local machine, OS keychain, browser session, or device authorization. The UX must expect partial success.

What Should Not Happen

No hidden dependency on the PC

If the runtime says Ready, closing the local PC must not kill access or running agent work.

No blind secret transfer

Raw .env, private keys, provider tokens, and auth folders require clear consent and should never pass through hosted collaboration services.

No silent project divergence

When local and remote checkouts differ, show the difference and require a choice before overwriting either side.

No fake readiness

A runtime is ready only after service, endpoint, auth, and remote API probes succeed.

UX Acceptance Criteria

  • A user can add a Linux VPS using only the desktop Connections page.
  • The resulting runtime remains reachable after the desktop app is closed.
  • The runtime appears in the composer runtime/checkout selector.
  • First remote use of a local project initializes a remote checkout automatically.
  • .camie/project-memory.md is included in project porting.
  • The UI distinguishes provider states: ready, copied, skipped, unavailable, needs login.
  • Secrets are never copied silently.
  • Failure states offer retry, repair, view logs, and remove runtime.

Section 2

How to Build It Exactly

This section maps the UX into implementation work inside the current repo shape. It keeps SSH launch, saved environments, provider settings, shared-project runtime concepts, and project memory aligned instead of introducing a separate remote-control system.

Architecture decision

The remote VPS is an execution environment. The desktop is a provisioning client and reconnecting UI. SSH is a setup channel, not the long-term transport.

Existing Code to Reuse

Area Current surface Reuse
Connections UI apps/web/src/components/settings/ConnectionsSettings.tsx Extend add-environment flow into persistent remote runtime wizard.
SSH bootstrap packages/ssh/src/tunnel.ts Reuse target parsing, auth prompting, Node fallback, remote runner install.
Saved environments apps/web/src/environments/runtime and client runtime stores Save remote runtime connection records and reconnect state.
Provider settings packages/contracts/src/settings.ts Sync provider instances/settings to remote runtime with explicit secret handling.
Shared runtime concepts packages/contracts/src/sharedProject.ts Reuse or align type names for ssh-vps runtime records.
Project CLI apps/server/src/cli/project.ts Register remote checkout after porting via existing project create path.
Update channel resolution packages/ssh/src/command.ts and desktop update settings Reuse release-channel logic so remote runtime installs follow t3@latest, t3@nightly, or a pinned release version.

New Concepts

PersistentRemoteRuntime

A saved remote execution target with SSH provisioning metadata, remote service metadata, endpoint metadata, provider health, and lifecycle state.

RemoteProjectPort

A mapping between a local project and its remote checkout on a specific runtime. This prevents repeated first-use sync and tracks divergence.

ProviderProfileTransfer

A structured transfer and verification flow for provider settings, homes, auth folders, and reauth requirements.

RemoteOwnedEndpoint

The URL or relay identifier that reaches the VPS without depending on the local desktop or its SSH tunnel.

Desktop IPC and backend APIs

Add desktop IPC methods rather than placing SSH/provisioning in the renderer. The renderer owns wizard state. Desktop main owns local process and SSH operations.

  • desktop:remote-runtime.preflight(target)
  • desktop:remote-runtime.provision(input)
  • desktop:remote-runtime.repair(runtimeId)
  • desktop:remote-runtime.remove(runtimeId, options)
  • desktop:remote-runtime.sync-provider-profiles(runtimeId, selection)
  • desktop:remote-runtime.prepare-project(runtimeId, localProjectRef)
  • desktop:remote-runtime.read-logs(runtimeId)

Add server or local API endpoints only for operations that need the current primary backend's project/settings state. Do not expose raw SSH credentials to the renderer after submission.

Provisioning Pipeline

1 Preflight

  • Resolve SSH target using existing packages/ssh helpers.
  • Connect with password/key and verify non-interactive sh.
  • Detect OS and architecture. MVP supports Linux x64 and arm64.
  • Check home directory write access and available disk.
  • Check or install git and curl through a bounded allowlist.
  • Check Node using existing remote Node fallback under ~/.t3/node.
  • Check service manager: prefer user systemd, fallback to managed background process with a warning.

2 Install KamiCode Runtime

  • Create ~/.kamicode/runtime for installer state.
  • Reuse ~/.t3/runner/t3-cli or align it under a new stable path.
  • Install package matching the selected update channel.
  • Run t3 --version remotely and save installed version.
  • Write a stable runner script owned by the SSH user with mode 700.
  • Write an update metadata file with channel, installed version, previous version, and package spec.

3 Create Persistent Service

The service must start without a GUI and keep the backend alive after SSH disconnects. 

[Unit]
Description=KamiCode remote runtime
After=network-online.target
Wants=network-online.target

[Service]
Environment=T3CODE_NO_BROWSER=1
ExecStart=%h/.kamicode/runtime/run-t3.sh serve --base-dir %h/.t3
Restart=always
RestartSec=5

[Install]
WantedBy=default.target
  • Use systemctl --user daemon-reload and enable --now.
  • Run loginctl enable-linger <user> only when available and permitted.
  • If user systemd is unavailable, fall back to nohup with a clear "less durable" warning.

4 Establish Remote-Owned Endpoint

This is the key difference from the current desktop-managed SSH tunnel. The endpoint has to survive local PC shutdown.

  1. Preferred: KamiCode Connect relay, using a headless/device-flow friendly link path.
  2. Supported alternative: Tailscale Serve if user has Tailscale configured on the VPS.
  3. Advanced alternative: user-provided HTTPS/WSS reverse proxy endpoint.
  4. Fallback only: SSH port forward, marked as Does not survive PC shutdown.

If KamiCode Connect currently requires browser OAuth on the remote host, add a desktop-assisted headless link flow before calling this feature complete. The desktop should authorize the relay, then the VPS should receive a scoped remote-runtime link credential over SSH.

5 Pair and Save Runtime

  • Run remote pairing command once the service is reachable.
  • Exchange pairing credential from the desktop.
  • Save environment/runtime record with endpoint metadata.
  • Probe server config, version, provider availability, and auth scopes.
  • Mark runtime Ready only after all required probes pass.

6 Remote Runtime Update Lifecycle

Updates should be performed by the desktop orchestrating SSH commands, or by a small remote updater command invoked through the running runtime. The remote service should not depend on the desktop staying online after the update is scheduled.

  1. Resolve target package from runtime channel: t3@latest, t3@nightly, or exact t3@x.y.z.
  2. Ask the remote runtime for current version, active turns, queued turns, service status, and previous version.
  3. If active work exists and policy is not forced, mark update as pending until idle.
  4. Install the new package into a versioned directory or update the runner cache atomically.
  5. Write new runner metadata, keeping the previous package spec and runner path.
  6. Restart the user service with systemctl --user restart or the configured fallback supervisor.
  7. Run health probes: HTTP, WebSocket, auth/session, server config, provider registry, and version.
  8. If probes pass, mark update complete and clear pending state.
  9. If probes fail, restore previous runner metadata, restart again, and mark rollback complete.

Protocol compatibility must be explicit. If the local desktop can talk to older remote servers, allow deferred updates. If schemas or RPC methods are incompatible, block new turns on that runtime and show Update required.

Project Porting Implementation

Git-aware path
  1. Local server resolves project: workspace root, repository identity, remote URL, branch, HEAD SHA, dirty state.
  2. Desktop asks remote runtime for existing project mapping.
  3. If missing, remote creates directory under ~/kamicode-projects/<slug>.
  4. Remote runs git clone using HTTPS or SSH URL depending on available auth.
  5. Remote checks out matching branch and attempts to reset to matching HEAD SHA.
  6. Local generates a patch for tracked dirty changes.
  7. Local packages selected untracked files, excluding denylisted files.
  8. Remote applies patch and overlays allowed untracked files.
  9. Remote registers project through its own server/project API.
  10. Save RemoteProjectPort mapping locally and remotely.
Archive sync fallback

Use this when the local project is not a git repo or has no usable remote. Implement packaging in Node so Windows desktop works without rsync.

  • Generate a manifest of files to copy.
  • Default include: source files, config, docs, package metadata, AGENTS/instructions, .camie/.
  • Default exclude: node_modules, build output, caches, .git, logs, screenshots, raw env files, private keys.
  • Show excluded secrets summary before first copy.
  • Stream tar/zip over SSH stdin to a remote unpack script.
  • Use checksums to avoid reuploading unchanged files on later syncs.
Required context files
  • .camie/project-memory.md must be copied when present.
  • AGENTS.md, CLAUDE.md, and other known instruction files should be copied when present.
  • Project package metadata and scripts must be copied so remote agents can infer setup.
  • Environment variable names may be copied, but secret values should not move without explicit consent.

Provider Profile Sync Implementation

Profile discovery

Add provider-specific profile descriptors. Each descriptor returns settings paths, optional auth paths, health probe commands, and transfer risk. 

ProviderProfileDescriptor {
  providerId: "codex" | "claude" | "opencode" | "cursor" | "grok" | "github"
  displayName: string
  settingsPaths: string[]
  authPaths: string[]
  defaultTransfer: "settings-only" | "ask" | "never"
  probeCommand: string[]
  redactedPreview: ProfilePreview
}
Transfer rules
  • Settings and provider instance configs can be copied by default.
  • Auth folders require explicit user selection.
  • Never print token file contents in logs.
  • Transfer over SSH only. Do not send provider secrets through hosted collaboration services.
  • After transfer, run remote provider probes and update provider health.
  • If probe fails, mark as Needs login and provide a guided reauth action.

  • Integrate with shared project/session snapshot workflows without silently publishing private local threads.

Definition of Done

  • Fresh VPS can be provisioned from the GUI with SSH credentials.
  • Remote runtime service restarts after VPS reboot and local desktop shutdown.
  • Remote-owned endpoint works without same-LAN or desktop SSH tunnel dependency.
  • Runtime appears in composer selector.
  • First remote send ports the local project and starts the turn afterward.
  • Provider profile sync reports accurate per-provider status.
  • Remote runtime updates can be checked, installed, restarted, and rolled back from the GUI.
  • Update policy respects active work and waits for idle unless the user forces the update.
  • Project secrets are excluded unless explicitly approved.
  • Automated tests cover wizard, provisioning events, project porting, and selector routing.

Do not ship as complete without this

The final runtime must have a remote-owned endpoint. If the only working path is an SSH tunnel owned by the local desktop process, the feature does not satisfy the core requirement.

This document intentionally uses "remote runtime" for the machine/session doing execution and "environment" for the backend connection or target app context. Keep those concepts separate during implementation.
